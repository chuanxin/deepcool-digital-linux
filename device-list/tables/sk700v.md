> Verified working on: **DeepCool AK series / SK700V** running on Linux (PVE / Debian).

## Key Differences from LQ / AK700 Series

The SK700V shares a similar HID packet structure with the LQ/ASSASSIN IV/AK700 series,
but has the following differences:

| Field | LQ / AK700 | **SK700V** |
|-------|-----------|-----------|
| D3 | 8 | **4** |
| D4 | 12 | **13** |
| D9 | Temperature Unit flag | **Power Percentage (progress bar)** |
| D10 | Temperature F32 [byte 0] | **Temperature Unit flag** |
| D11~D14 | — | **Temperature F32 (4 bytes)** |
| Checksum range | D1~D16 | **D1~D17** |
| Termination byte | D18 | **D19** |

## Notes on Power Fields

### D7~D8: Power Consumption (Big-Endian U16)

The firmware reads power as a **Big-Endian 16-bit unsigned integer**:

```
displayed_watts = (D7 << 8) | D8
```

The original implementations incorrectly set D7=0 and encoded power as Little-Endian in D8~D9.
This worked by accident for power values below 256W (where the high byte is 0), but would cause
display corruption if D7 was ever set to a non-zero value.

### D9: Power Percentage (Progress Bar)

The SK700V display shows a progress bar beneath the wattage value.
D9 (0–100) controls its fill level independently of the actual wattage in D7~D8.

**Recommended**: derive from RAPL PL2 (short-term burst limit):
```
D9 = clamp((current_watts / pl2_watts) * 100, 0, 100)
```

On Linux, PL2 can be read from:
```
/sys/class/powercap/intel-rapl:0/constraint_1_power_limit_uw
```

## Initialization Packet

Before the main loop, send a one-time init packet:

```
D0 = 16, D1 = 170, D2~D63 = 0
```
